# План реализации «Коридор» (онлайн)

Веб-игра Quoridor 9×9 для игры двух людей онлайн через ссылку-приглашение.

## Решения (зафиксировано)

- **Без аккаунтов** — игра по ссылке-приглашению (комната по коду).
- **Хранилище — PostgreSQL** — партии переживают перезапуск сервера, есть история.
- **Realtime — WebSockets** (Flask-SocketIO): соперник видит ход сразу.
- **Сервер хранит истину** о состоянии; клиенту не доверяем.
- **Режимы (MVP):** лобби с открытыми комнатами, игра по ссылке-приглашению.
  Игра с ИИ — **вне MVP** (см. «Замечания»), добавляем позже.
- **Хостинг — PaaS (Railway/Render)**: сервер работает 24/7 без участия автора.

---

## Архитектура

```
Браузер A ─┐                     ┌─ Flask (HTTP: страницы, создание/вход в комнату)
           ├── WebSocket ──► Сервер ┤
Браузер B ─┘                     ├─ Flask-SocketIO (ходы, рассылка состояния)
                                 ├─ game/  (чистая логика, тестируемая)
                                 └─ PostgreSQL (партии, ходы, состояние)
```

### Слои бэкенда

1. **`game/` — игровое ядро (чистый Python, без Flask).**
   Доска 9×9, позиции фишек, стены, легальность ходов, проверка «стена не
   перекрывает путь полностью» (BFS для каждого игрока), определение победы.
   Полностью покрыто юнит-тестами, не зависит от веба и БД.

2. **HTTP-слой (Flask).**
   Главная страница, создание игры, вход по ссылке/коду комнаты, отдача статики.

3. **Realtime-слой (Flask-SocketIO).**
   События: подключение к комнате, ход (фишка/стена), рассылка обновлённого
   состояния обоим игрокам, уведомления (ход соперника, победа, дисконнект).

4. **Слой данных (PostgreSQL).**
   Хранение партий и состояния. Доступ через SQLAlchemy (ORM) либо psycopg
   напрямую. Миграции — Alembic.

### Фронтенд

- HTML + **Canvas** + ванильный JS (без React — для одной доски избыточно).
- Canvas рисует доску, фишки, стены, подсветку доступных ходов.
- JS-клиент держит WebSocket, шлёт ходы, получает состояние, перерисовывает.

## Модель данных (PostgreSQL, черновик)

```
games
  id            uuid (pk)          -- он же код комнаты в ссылке
  status        text               -- waiting | active | finished | abandoned
  mode          text               -- ai | online
  private       boolean            -- true = только по ссылке, не в лобби
  state         jsonb              -- текущее состояние доски (источник истины)
  player1_token uuid               -- секрет в куке/URL, чтобы вернуться за свою сторону
  player2_token uuid
  turn          smallint           -- чей ход: 1 или 2
  winner        smallint null
  created_at    timestamptz
  updated_at    timestamptz

moves                              -- история (для реплея/отладки), опционально на старте
  id        bigserial (pk)
  game_id   uuid (fk games.id)
  ply       int                    -- номер полухода
  player    smallint
  action    jsonb                  -- {type: move|wall, ...}
  created_at timestamptz
```

Идентификация игрока без логина: при создании/входе выдаём `player_token`
(в куке или в URL), по нему сервер понимает, за какую сторону играет клиент.

## Поток игры (happy path)

1. Игрок A открывает сайт → «Создать игру» → создаётся `games` (status=waiting),
   A получает ссылку вида `/game/<id>` и `player1_token`.
2. A делится ссылкой. Игрок B открывает её → занимает вторую сторону,
   получает `player2_token`, status → active.
3. Оба подключаются к SocketIO-комнате `<id>`.
4. Чей ход — тот шлёт `action`. Сервер: грузит state из БД → проверяет
   легальность в `game/` → применяет → пишет в БД → рассылает новое state обоим.
5. При победе: status=finished, winner проставлен, рассылка финала.
6. Реконнект: клиент по `player_token` снова входит в комнату, получает
   актуальный state из БД.

## Режимы игры

1. **Игра с ИИ** — бот считается прямо на сервере (отдельная инфраструктура не
   нужна). Игрок vs `game/`-логика + алгоритм бота. Не требует второго человека.
2. **Лобби с открытыми комнатами** — экран со списком игр `status=waiting`.
   Игрок видит ждущих соперников и заходит в любую. Это та же модель `games`,
   плюс список + кнопка «войти».
3. **Игра по ссылке** — приватная комната: создал → отправил ссылку другу.
   Такая игра в общем лобби не показывается (флаг `private`).

> Все три режима — **один и тот же сервер и одна база**, отличается только вход
> в партию. Отдельных серверов под бота или лобби не нужно.

## Хостинг / запуск

- **Локально:** `python app.py` (SocketIO dev-сервер), Postgres в Docker.
- **Прод:** gunicorn + worker с поддержкой WebSocket (eventlet/gevent),
  управляемый Postgres. Платформы: Railway / Render / Fly.io
  (бесплатные тиры, есть Postgres и WebSockets).
- Конфиг через переменные окружения (`DATABASE_URL`, `SECRET_KEY`).

### Как устроен деплой (PaaS, без покупки железа)

Сервер не покупается, а арендуется как услуга. Схема на Railway/Render:

1. Код лежит в GitHub-репозитории.
2. Подключаешь репозиторий к Railway/Render — платформа сама собирает и
   запускает приложение при каждом `git push`.
3. Добавляешь PostgreSQL «в один клик» — платформа выдаёт `DATABASE_URL`.
4. Прописываешь переменные окружения (`DATABASE_URL`, `SECRET_KEY`).
5. Получаешь публичный URL (`https://corridor.up.railway.app`) — игроки заходят
   по нему в любой момент, твой компьютер при этом выключен.

Стоимость на старте — практически 0 ₽ (бесплатные тиры), платишь только при
росте нагрузки. Деплой — это **этап 7**, после рабочего локального прототипа.

## Этапы (предлагаемый порядок)

1. **Игровое ядро `game/` + тесты.** Доска, ходы, стены, BFS-проверка, победа.
   Не зависит ни от чего — фундамент.
2. **Инфраструктура БД.** Postgres локально (Docker), модели, миграции, `db.py`.
3. **HTTP-скелет.** Создание игры, вход по ссылке, выдача токенов, страница доски.
4. **Canvas-рендер.** Отрисовка статичного состояния, подсветка ходов.
5. **Realtime.** SocketIO: комнаты, приём хода, валидация ядром, рассылка state.
   Сделать **единый код-путь записи хода** — одна транзакция под `SELECT … FOR UPDATE`
   на строке партии, которая атомарно: загружает `state`, валидирует ядром,
   **переприсваивает `state` целиком** (`game.state = new.to_json()`; мутации JSONB
   «на месте» SQLAlchemy не трекает) и в той же транзакции обновляет колонки-дубли
   `ply`/`winner`/`status` + пишет строку в `moves` (`UNIQUE(game_id, ply)` против
   дублей). Никаких других мест записи хода — чтобы дубли-колонки не разъезжались
   с истиной в `state`.
6. **Полный онлайн-цикл.** Реконнект, обработка дисконнекта, экран победы.
7. **Деплой.** gunicorn + eventlet, managed Postgres, env-конфиг.

## Замечания (review)

1. **`eventlet` — риск.** Полу-заброшен, плохо дружит со свежим Python.
   По умолчанию рассмотреть **`gevent` + `gevent-websocket`** как альтернативу.
   Решение принять на этапе 5/7.
2. **Конкурентность — защита от гонок.** Два игрока, общее состояние в БД.
   При «одновременных» ходах нужна защита: `SELECT ... FOR UPDATE` на строке
   партии **или** проверка номера полухода `ply` (отвергать ход с устаревшим
   `ply`). Использовать `moves.ply` как защиту от гонок и дублей. Этап 5/6.
3. **Режим с ИИ — вне MVP.** Сначала человек-vs-человек. Бот для «Коридора»
   нетривиален (минимакс + оценка позиции) — отдельный крупный объём, добавляем
   после рабочего онлайн-цикла.
4. **Тонкий вертикальный срез раньше.** До полноценного Canvas сделать сквозную
   цепочку «ход → обновление состояния» хоть в текстовом виде, чтобы realtime
   заработал раньше, чем вкладываться в красивый рендер.
5. **Безопасность токенов.** `player_token` — секрет. Кука `HttpOnly` (и `Secure`
   в проде). Токен соперника никогда не должен утекать клиенту в лобби или в
   публичном состоянии партии.
